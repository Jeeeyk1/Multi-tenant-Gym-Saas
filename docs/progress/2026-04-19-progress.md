### Endpoints

```
POST   /gyms/:gymId/staff                          → invite staff
DELETE /gyms/:gymId/staff/:staffId                 → deactivate staff
POST   /gyms/:gymId/staff/:staffId/roles           → assign role
DELETE /gyms/:gymId/staff/:staffId/roles/:roleId   → remove role
GET    /gyms/:gymId/staff                          → list staff
```

All endpoints are protected by `JwtAuthGuard`. Permission + gym-scope checks are enforced inside each use case.

### Key decisions

- **Invite existing user**: If the invited email already belongs to a user account, skip user creation and link the existing account to gym_staff. Only the gym_staff + invite token are created.
- **Invite token type**: Stored as `type = 'STAFF_INVITE'` in `user_tokens`. MVP returns the raw token in the response. Production: relay via email.
- **Permission check pattern**: Gym-level callers must have `staff.manage` for write operations (`invite`, `deactivate`, `assign-role`, `remove-role`) and `staff.view` or `staff.manage` for list. Org-level callers are trusted (same MVP pattern as Phase 4).
- **Gym scope check**: Gym-level callers with a JWT gymId that doesn't match the route `:gymId` are rejected with `ForbiddenError`. Org-level callers pass through.
- **Duplicate role assignment**: Checked before inserting via `findStaffRoleAssignment`. Throws `ConflictError` if already assigned — avoids relying on DB unique constraint for user-facing errors.
- **Role removal**: Validates that the role assignment exists before attempting delete, so a clean `NotFoundError` is returned instead of a Prisma exception.
- **list-staff permission**: Accepts `staff.view` OR `staff.manage` — a manager can list staff without needing a separate view-only permission.

### Endpoints

```
POST   /gyms/:gymId/plans              → create plan (gym.settings)
PATCH  /gyms/:gymId/plans/:planId      → update plan (gym.settings)
GET    /gyms/:gymId/plans              → list plans (members.view)

POST   /gyms/:gymId/members            → register member (members.create)
GET    /gyms/:gymId/members            → list members paginated (members.view)
GET    /gyms/:gymId/members/:id        → member detail (members.view)
GET    /gyms/:gymId/members/:id/qr     → QR token (members.view or own)
PATCH  /gyms/:gymId/members/:id/suspend     → suspend (members.suspend)
PATCH  /gyms/:gymId/members/:id/reactivate → reactivate (members.edit)
```

### Key decisions

- **Plan type**: Free-form string (e.g. "MONTHLY", "ANNUAL"). Not enum-restricted since the docs don't specify allowed values.
- **Expiry resolution**: If `planId` given → expiryDate = today + plan.durationDays. If no planId → `expiryDate` field required in DTO. Use case throws `MISSING_EXPIRY` if neither provided.
- **Registration always creates new user**: If email already exists → ConflictError(`EMAIL_TAKEN`). Consistent with "one account per gym" rule from auth doc.
- **membership_number format**: `MBR-` prefix + 8 random chars from safe charset (no ambiguous chars). Globally unique.
- **qr_code_token**: 32 random bytes as hex (64 chars). Globally unique.
- **Registration transaction**: user → gym_member → member_privacy → conversation_member (default conversation if exists) → invite token (MEMBER_INVITE). All in one Prisma $transaction.
- **Suspension scope**: Only sets gym_members.status. Does not touch users.is_active. Per domain docs, suspension is gym-scoped only.
- **Reactivate**: Always sets status=ACTIVE (even from EXPIRED). Staff-driven manual override.
- **QR access**: Staff with `members.view` can access any member's QR. A member (gym JWT) can only access their own QR (userId match).
- **async execute methods**: All use case `execute` methods must be async so that synchronous guard throws become rejected promises (required for `.rejects` in Jest tests).
