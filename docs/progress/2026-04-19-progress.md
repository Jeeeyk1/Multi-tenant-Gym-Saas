### Endpoints

```
POST   /gyms/:gymId/staff                          → invite staff
DELETE /gyms/:gymId/staff/:staffId                 → deactivate staff
POST   /gyms/:gymId/staff/:staffId/roles           → assign role
DELETE /gyms/:gymId/staff/:staffId/roles/:roleId   → remove role
GET    /gyms/:gymId/staff                          → list staff
```

All endpoints are protected by `JwtAuthGuard`. Permission + gym-scope checks are enforced inside each use case.

### Key decisions

- **Invite existing user**: If the invited email already belongs to a user account, skip user creation and link the existing account to gym_staff. Only the gym_staff + invite token are created.
- **Invite token type**: Stored as `type = 'STAFF_INVITE'` in `user_tokens`. MVP returns the raw token in the response. Production: relay via email.
- **Permission check pattern**: Gym-level callers must have `staff.manage` for write operations (`invite`, `deactivate`, `assign-role`, `remove-role`) and `staff.view` or `staff.manage` for list. Org-level callers are trusted (same MVP pattern as Phase 4).
- **Gym scope check**: Gym-level callers with a JWT gymId that doesn't match the route `:gymId` are rejected with `ForbiddenError`. Org-level callers pass through.
- **Duplicate role assignment**: Checked before inserting via `findStaffRoleAssignment`. Throws `ConflictError` if already assigned — avoids relying on DB unique constraint for user-facing errors.
- **Role removal**: Validates that the role assignment exists before attempting delete, so a clean `NotFoundError` is returned instead of a Prisma exception.
- **list-staff permission**: Accepts `staff.view` OR `staff.manage` — a manager can list staff without needing a separate view-only permission.

```
GET    /gyms/:gymId/members/:id/qr     → QR token (members.view or own)
PATCH  /gyms/:gymId/members/:id/suspend     → suspend (members.suspend)
PATCH  /gyms/:gymId/members/:id/reactivate → reactivate (members.edit)
```

### Key decisions

- **Plan type**: Free-form string (e.g. "MONTHLY", "ANNUAL"). Not enum-restricted since the docs don't specify allowed values.
- **Expiry resolution**: If `planId` given → expiryDate = today + plan.durationDays. If no planId → `expiryDate` field required in DTO. Use case throws `MISSING_EXPIRY` if neither provided.
- **Registration always creates new user**: If email already exists → ConflictError(`EMAIL_TAKEN`). Consistent with "one account per gym" rule from auth doc.
- **membership_number format**: `MBR-` prefix + 8 random chars from safe charset (no ambiguous chars). Globally unique.
- **qr_code_token**: 32 random bytes as hex (64 chars). Globally unique.
- **Registration transaction**: user → gym_member → member_privacy → conversation_member (default conversation if exists) → invite token (MEMBER_INVITE). All in one Prisma $transaction.
- **Suspension scope**: Only sets gym_members.status. Does not touch users.is_active. Per domain docs, suspension is gym-scoped only.
- **Reactivate**: Always sets status=ACTIVE (even from EXPIRED). Staff-driven manual override.
- **QR access**: Staff with `members.view` can access any member's QR. A member (gym JWT) can only access their own QR (userId match).
- **async execute methods**: All use case `execute` methods must be async so that synchronous guard throws become rejected promises (required for `.rejects` in Jest tests).

### Endpoints

```

GET    /gyms/:gymId/checkins/active              → who is currently in gym (checkins.view)
GET    /gyms/:gymId/checkins                     → history paginated (checkins.view)
PATCH  /gyms/:gymId/checkins/:id/checkout        → manual checkout

POST   /gyms/:gymId/members/:id/renew            → process renewal (members.renew)
GET    /gyms/:gymId/members/:id/renewals         → renewal history (members.view)
```

### Key decisions

- **Schedule validator is a pure domain function** accepting `(gym, now, timezone)` — `now` injected as a parameter so tests can control time without mocking the clock.
- **Timezone handling**: Uses `Intl.DateTimeFormat` with `hourCycle: 'h23'` to extract local day-of-week and time-of-day in the gym's timezone. No external library required.
- **Out-of-hours is a flag, not a block**: Check-in proceeds regardless; `is_out_of_hours` is set on the record for staff review. 24/7 gyms always get `false`.
- **Stale auto-close**: If an open check-in is older than `gym.auto_checkout_hours`, it is auto-closed (`is_auto_checkout = true`) before creating the new check-in. Fresh open check-ins are rejected with `ALREADY_CHECKED_IN`.
- **processedBy**: Set to caller.sub for staff methods (MANUAL_STAFF, QR_STAFF_SCAN); null for self methods (QR_SELF_SCAN, APP_SELF_CHECKIN).
- **Checkout permissions**: Staff with `checkins.manage` can check out any member. A gym-level member (JWT) can check out their own active check-in by matching memberId via their userId.
- **Renewal expiry extension**: ACTIVE members with future expiry get durationDays added to their current expiry date. EXPIRED/SUSPENDED members get durationDays added from today. Both reset status to ACTIVE.
- **Plan fallback on renewal**: If no planId in the request, the use case reuses the member's existing `membershipPlanId`. If no plan exists either, `MISSING_PLAN_ID` is returned.

### Key decisions

- **Cron config file**: All schedule expressions and lock TTLs centralised in `cron.config.ts`. Job classes import constants, not raw strings.
- **Distributed lock**: `CronLockService.withLock(key, ttlSeconds, fn)` — acquires via Redis SET NX EX. If not acquired, logs a skip message and returns immediately without running the job body.
- **Batch processing (MembershipExpiryJob)**: `findMany({ take: BATCH_SIZE })` → `updateMany({ id: { in: [...] } })` → repeat until empty. Prevents memory spikes on large datasets.
- **Per-gym threshold filtering (AutoSuspendJob, AutoCheckoutJob)**: Records are loaded with their gym config included. Application code filters which records exceed the per-gym threshold. Then a single `updateMany` on the filtered IDs. Prisma cannot express per-row join conditions in `where`.
- **AnnouncementPublisherJob**: Uses two direct `updateMany` calls (SCHEDULED→PUBLISHED, PUBLISHED→EXPIRED) — no per-record filtering needed since the DB `where` clause handles the time comparison.
- **jest.resetAllMocks() in cron specs**: All 4 cron spec files use `resetAllMocks()` in `beforeEach`, not `clearAllMocks()`. `clearAllMocks()` only clears call history — it does NOT clear `mockResolvedValueOnce` queues, causing bleed between tests.

### Endpoints

```
POST   /gyms/:gymId/announcements        → create (announcements.manage)
PATCH  /gyms/:gymId/announcements/:id    → update (announcements.manage)
DELETE /gyms/:gymId/announcements/:id    → archive (announcements.manage)
GET    /gyms/:gymId/announcements        → list (staff: all; members: PUBLISHED only)
POST   /gyms/:gymId/announcements/:id/read → mark as read
```

### Key decisions

- **Status resolution on create/update**: `publishAt` null or past → PUBLISHED immediately; `publishAt` future → SCHEDULED. No explicit DRAFT creation via API — cron job handles the SCHEDULED→PUBLISHED transition.
- **ARCHIVED vs EXPIRED**: Both are terminal for editing. EXPIRED/ARCHIVED announcements throw `ANNOUNCEMENT_NOT_EDITABLE` on update. Archive is blocked if already EXPIRED or ARCHIVED.
- **Archivable statuses**: DRAFT, SCHEDULED, PUBLISHED may all be archived manually. EXPIRED cannot — it is a system-managed terminal state.
- **List visibility**: Callers with `announcements.manage` permission (or org-level) see all statuses and can optionally filter by status query param. All others see only PUBLISHED.
- **Read tracking**: `upsert` on `(announcement_id, user_id)` unique index — idempotent, no error on double-read.
- **markRead only works on PUBLISHED**: If announcement is not found or not PUBLISHED, returns `ANNOUNCEMENT_NOT_FOUND` — avoids revealing status to unauthorized callers.
- **orderBy**: isPinned DESC → publishAt DESC → createdAt DESC — pinned announcements always surface first.
