## Endpoints delivered

```
GET    /gyms/:gymId/conversations/:id/messages        → paginated feed (limit + before cursor)
POST   /gyms/:gymId/conversations/:id/messages        → send message (201)
DELETE /gyms/:gymId/conversations/:id/messages/:msgId → soft delete (200)
POST   /gyms/:gymId/conversations/:id/messages/:msgId/react → toggle emoji (200)
POST   /gyms/:gymId/conversations/:id/read            → mark conversation read (200)
```

## Key decisions

- **Membership check for send**: Repository looks up `gym_members` for the calling user. If a record exists and status ≠ ACTIVE → `MEMBERSHIP_NOT_ACTIVE`. If no record (staff) → allowed. This cleanly handles the staff vs member distinction without relying on permission checks.
- **Soft delete**: Content is wiped (`content: null`), `is_deleted = true`. Deleted messages are still returned by list so clients can render "This message was deleted". No hard deletes.
- **Reaction toggle**: Single endpoint (`POST .../react`) — if the reaction exists, it is removed and `{ action: 'removed' }` is returned; if absent, it is added and `{ action: 'added' }` is returned.
- **Pagination**: Cursor-based via `before` (ISO-8601 sentAt timestamp) + `limit` (max 100, default 50). Messages returned in `sentAt DESC` order. Suitable for infinite-scroll chat UIs loading older messages.
- **`last_read_at` strategy**: Uses `conversation_members.last_read_at` for community channels (efficient for large channels). Per-message `message_reads` table exists in schema for future direct/group chat use but is not used in this MVP phase.
- **Org token guard**: `requireGymUser()` helper in controller throws `GYM_ACCESS_DENIED` immediately for `type: 'org'` tokens. Chat is gym-scoped only.

## WebSocket migration (same session)

Migrated chat from REST-only to real-time WebSocket with Redis pub/sub.

### Added
- `apps/api/src/common/adapters/redis-io.adapter.ts` — custom IoAdapter wrapping ioredis pub/sub
- `apps/api/src/modules/chat/chat.gateway.ts` — `/chat` namespace gateway
  - `handleConnection` — JWT auth on handshake, rejects org tokens
  - `join` — validates conversation membership before allowing room subscription
  - `message.send` — persists via use case, broadcasts `message.new` to room
  - `message.delete` — soft-deletes via use case, broadcasts `message.deleted` to room
  - `message.react` — toggles via use case, broadcasts `message.reaction` to room
  - `conversation.read` — updates `last_read_at`, no broadcast (per-user state)
- Updated `apps/api/src/modules/chat/chat.module.ts` — added `ChatGateway`
- Updated `apps/api/src/main.ts` — `RedisIoAdapter` registered before `app.listen()`

### Packages added
- `@nestjs/websockets@^10`
- `@nestjs/platform-socket.io@^10`
- `socket.io`
- `@socket.io/redis-adapter`

### Key decisions
- Room naming `conversation:{id}` works identically for COMMUNITY, DIRECT, and GROUP — no gateway changes needed when those types are added
- Redis adapter enables horizontal scaling without application-level changes
- REST endpoints kept — still needed for message history loading and Postman testing
- Use cases remain pure — gateway calls them and handles the broadcast, keeping business logic decoupled from transport
