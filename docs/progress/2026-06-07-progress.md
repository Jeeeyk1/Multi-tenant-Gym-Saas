## Key Decisions
- **`authClient` vs `api`**: `authClient` is an unauthenticated axios instance for login/resolve-code/logout. `api` creates a per-request authenticated instance by reading the httpOnly cookie each time (required for Server Components / Server Actions).
- **`GET /staff/roles` endpoint**: Added to the staff module to support role assignment UI; roles have UUID primary keys so the UI can't construct them from names alone.
- **Org portal JWT**: `organizationId` and `orgRole` were already in the org JWT payload but not exposed by `getSessionUser()` — added both fields.
- **Mobile screens**: All M-M1–M-M5 were already built in a prior session (not captured in last progress log).
- **WebSocket for chat**: The API already had a full Socket.IO gateway (`/chat` namespace). Web chat now uses it directly. JWT token is read server-side (httpOnly cookie) and passed as a prop to the client component for the socket handshake.
- **Gym theme migration**: `primary_color` and `logo_url` were already in `0005_gym_profile_schedules_features.sql` — no separate theme migration needed. The noted `0017_gym_theme.sql` was never required.
- **`/me/profile` vs `/:memberId/profile`**: Member's own profile uses `/me/profile` (no extra permissions). Staff viewing/editing any member's profile uses `/:memberId/profile` with `members.view`/`members.edit`.

```
AI_MODEL=grok-3             # optional; defaults: grok-3 / claude-sonnet-4-6
```
