### Phase 19 — Food Photo Analysis (Claude Vision)

**Goal:** Members take a photo of their food, Claude Vision analyses it and returns macros (calories, protein, carbs, fat). Result can be saved to the food diary. Text-based meal analysis remains fully intact as a fallback when no photo is provided.

---

### Architecture Decision

Two separate AI providers running simultaneously:

| Token | Provider | Purpose | Config |
|-------|----------|---------|--------|
| `AI_PROVIDER_PORT` | Groq (existing) | Text AI — workout suggestions, meal suggestions, text meal analysis | `AI_API_KEY`, `AI_MODEL` |
| `VISION_PROVIDER_PORT` | Anthropic Claude (new) | Vision — food photo analysis only | `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL` |

Neither interferes with the other. Text AI features continue to use Groq unchanged.

### Key Decisions

- **Claude Haiku as default vision model**: ~$0.002/call vs ~$0.008 for Sonnet. More than capable for food recognition. Configurable via `ANTHROPIC_MODEL` env var if accuracy needs to improve.
- **Upload-then-analyse (two steps)**: Photo goes to Cloudinary first, then the URL is passed to Claude. Avoids base64 encoding overhead and stores the photo permanently so it can be shown in the food diary later.
- **Text analysis unchanged**: Existing text-describe-your-meal flow still works. `AnalyseMealUseCase` branches on whether `photoUrl` is present — no breaking change.
- **No "ask for clarification" round-trip**: Claude's `notes` field surfaces any uncertainty (e.g. "portion size unclear"). User can add a manual note before saving but there's no second AI call. Keeps cost predictable.
- **`ANTHROPIC_API_KEY` separate from `AI_API_KEY`**: Intentional — lets you run Groq for all text features and only pay Claude for vision, without any toggle switch.

## Phase 19 (cont.) — Exercise GIF with DB-First Cache + ExerciseDB Fallback

### What Was Built

When a member taps an exercise card and the detail modal opens, the app now tries to show an animated GIF of the exercise. The GIF is sourced from ExerciseDB (RapidAPI), but only fetched once per exercise — subsequent requests hit the local DB cache.

### Architecture

```
GET /ai/exercise-media?name=<exerciseName>
  → check exercise_media table (normalized_name = exerciseName.toLowerCase())
      ↓ HIT → return { gifUrl }
      ↓ MISS → call ExerciseDB RapidAPI → store in DB → return { gifUrl }
```

If `EXERCISEDB_API_KEY` is not set, the adapter logs a warning and returns `null` (no GIF, no crash).

### DB Changes

**`db/migrations/0025_exercise_media.sql`** (new):
```sql
CREATE TABLE exercise_media (
  id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  normalized_name TEXT        NOT NULL,
  gif_url         TEXT        NOT NULL,
  fetched_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CONSTRAINT uq_exercise_media_name UNIQUE (normalized_name)
);
```
