  - Added `findUserWithFirstGymMember(userId)` — looks up the member's gym name and code for the welcome email

### 3. Updated ActivateAccountUseCase

  - `INVITE` tokens (staff) → `sendStaffWelcomeEmail` (existing flow, renamed)
  - `MEMBER_INVITE` tokens (members) → `sendMemberWelcomeEmail` (new)
  - Both are fire-and-forget; failures are logged, not surfaced to the caller

### 4. Updated activate-account spec
- `apps/api/src/modules/identity/application/use-cases/activate-account.use-case.spec.ts`
  - Added `findUserWithFirstGymMember: jest.fn()` and `sendMemberWelcome: jest.fn()` to mocks
  - Added test: "sends member welcome email for MEMBER_INVITE token"

### 5. Fixed web activation UX
**Before:** After activating, member was redirected to `/{code}/login?activated=1` (staff login page) which told them to "Sign in below". When they signed in, they hit `/not-staff` — confusing mixed signals.

**After:**
- `apps/web/src/app/member-activate/actions.ts` — returns `{ success: true }` on success instead of calling `redirect()`
- `apps/web/src/app/member-activate/activate-form.tsx` — shows an inline success state with:
  - Green checkmark
  - "You're all set!" heading
  - Gym code displayed prominently in monospace
  - Instructions to open the app, enter the gym code, and sign in

## Key decisions

- Members use `MEMBER_INVITE` token type; staff use `INVITE` — this enables the use case to send the right post-activation email without an extra DB lookup to infer role
- Web activation now stays on the activation page after success (no redirect to staff portal), preventing the confusing "sign in below" → "use the mobile app" loop
- The member welcome email sends the gym code in a styled code block so the member can easily find it when opening the mobile app

## HealthKit — production-ready fix

### Problem
`react-native-healthkit@0.0.1` was a stub package with no code. The `require()` in `healthkit.service.ts` was silently failing inside the try/catch, so HealthKit data was always `null` even on iOS.

### What was done
- Removed `react-native-healthkit@0.0.1` stub
- Installed `@kingstinct/react-native-healthkit@^14.0.2` (the actively maintained package)
- Updated `apps/mobile/src/services/healthkit.service.ts`:
  - Changed `require('react-native-healthkit')` → `require('@kingstinct/react-native-healthkit')`
  - Updated `requestAuthorization` call: now uses `{ toRead: [...] }` shape (v14 API)
  - Updated `queryQuantitySamples` options: date range is now `filter: { date: { startDate, endDate } }` with required `limit: -1`
  - Identifiers are now plain string literals (`'HKQuantityTypeIdentifierActiveEnergyBurned'`, `'HKQuantityTypeIdentifierHeartRate'`) instead of enum members
- `app.json` `ios.infoPlist` already had `NSHealthShareUsageDescription` and `NSHealthUpdateUsageDescription` — no change needed
- Android: still safely skipped via `Platform.OS !== 'ios'` guard

### Key decisions
- Using string literal identifiers directly (not an enum) — v14 of the package uses TypeScript union string types
- Date filter moved into `filter.date` nested object per v14 query API
- No Android Health Connect added — deferred; the Platform guard handles it cleanly
