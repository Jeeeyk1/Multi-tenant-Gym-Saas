## Key decisions

- **TanStack Query sits on top of axios/fetch — it does not replace them.** Existing HTTP clients (`apps/web/src/lib/api.ts`, `apps/mobile/src/services/api.ts`) stay; auth refresh logic stays in the fetcher layer where it already works.
- **Mobile first, web second.** Mobile has zero caching today, so every migrated screen is an immediate UX win. Web is mostly server components where TanStack Query adds nothing, so the web-side scope is narrow (only `"use client"` components and hybrid pages via `HydrationBoundary`).
- **Phase 0 (decisions) before Phase 1 (code).** Locking query-key shape and `QueryClient` defaults now is cheap; retrofitting across 50 hooks later is painful.
- **Services stay.** `apps/mobile/src/services/*.ts` remain as the thin transport layer — they become the body of `queryFn` / `mutationFn`. Hooks don't inline `fetch`.

### Visible win
`getMyMember` is now deduped across profile and dashboard. Tabbing between them no longer re-fetches the member.

### Patterns locked in
- Hooks read `gymId` from `useAuth()` — screens never pass it.
- 404 → null/[] handled inside `queryFn` (for "no profile yet" semantics). All other errors flow to `query.error`.
- Mutations write the server response straight to cache via `setQueryData`. `invalidateQueries` reserved for cross-resource invalidation.
- Mutation hooks return the full `useMutation` result — never wrapped in a custom shape.
- Pull-to-refresh = `query.isRefetching` + `query.refetch()`. No `useState`.

- `(member)/checkin.tsx` — uses `useSelfCheckIn`

### Intentionally not migrated
- **`auth.service.ts`** — login / refresh / logout are command-style operations owned by `AuthContext`; they don't benefit from TanStack Query caching.
- **`(member)/chat.tsx`** — messages are WebSocket-driven (socket.io). The two HTTP calls (`getDefaultConversation`, `getMessages`) seed initial state once per session and never refetch. Forcing them into TanStack Query would either let stale cache fight live WS events or require cache merging on every event. Not worth the complexity.

### Patterns confirmed in practice
- The rule "read `gymId` from `useAuth()` inside the hook" scales well — no `gymId` props are threaded through these 14 migrated screens.
- Optimistic mutations with `onMutate` → `onError` rollback work cleanly for list-removal (check-out, approve/reject, archive) and field-update (announcement mark-read).
- For chained mutations (leaderboard photo upload → PR submit), `mutateAsync` from two separate hooks composes correctly while letting the screen show interim progress.
- `useInfiniteQuery` is the right shape for paginated history; `flatMap(p => p.data)` flattens pages back into a single list for the UI.
- When two screens share a query (dashboard's `useMyMember` and profile's `useMyMember`), tab-switching is now a cache hit on each side.
