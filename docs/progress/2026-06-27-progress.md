## Key decisions

- **`react-hooks/exhaustive-deps` is `warn`, not `error`.** RN animation patterns (stable `Animated.Value` refs in `useEffect(..., [])`) trigger the rule but are correct. Erroring would force noisy `eslint-disable` comments everywhere.
- **`no-restricted-syntax` warns instead of errors** on `useEffect+setState` patterns. Setter naming heuristic (`setData`, `setItems`, `setList`, `setResults`, `setMembers`, `setSessions`, `setAnnouncements`, `setWorkouts`) catches the bad shape without blocking legitimate UI-state effects.
- **Chat keeps its console logs.** WebSocket connect/disconnect/error traces are still useful in dev. Marked with a file-level disable so the intent is explicit.

## Update — Badges UX: equip flow + chat/leaderboard surfacing

Backend had badges since two weeks ago, but they only appeared as a list on the profile screen. Members had no way to choose a "main" badge, and no one else could see what badges other members had earned. This update ships the equip flow and surfaces the equipped badge next to author names everywhere it matters.

### API — new endpoints and use cases
- `PATCH /gyms/:gymId/badges/my/:badgeId` body `{ equipped: boolean }` — toggles `is_equipped` on the caller's badge. When equipping, atomically unequips any other equipped badge so each member has exactly 0–1 equipped at a time. Refuses to equip an expired badge but allows unequipping.
- `GET /gyms/:gymId/badges/equipped` — returns `{ userId, badge }[]` for all active members in the gym who have a badge equipped. Excludes expired. Designed for clients to build a `Map<userId, EquippedBadge>` once and consult it while rendering author rows.
- New use cases: `EquipBadgeUseCase`, `ListEquippedBadgesUseCase`. Wired through `BadgesController` and `BadgesModule`.
- `BadgesRepository` gained three methods: `findMemberBadge`, `setBadgeEquipped` (transactional unequip-other + equip-this), `listEquippedBadgesByGym`.
- DTO: `EquipBadgeDto` validates the `equipped: boolean` body.
- `LeaderboardRepository` query expanded to also select `member.user.id` so the client can join leaderboard rows to the equipped-badges map. Type updated on the mobile `LeaderboardEntry`.

### Mobile — surfacing
- `apps/mobile/src/components/EquippedBadgeChip.tsx` (new) — small ringed icon component used in chat sender row and leaderboard rank row. Optional name label for richer surfaces later.
- `apps/mobile/src/hooks/members.ts` — added `useEquippedBadgesMap()` (returns `Map<userId, EquippedBadge>` with 60s staleTime) and `useEquipBadge()` mutation that invalidates both my-badges and equipped-badges queries.
- `apps/mobile/src/app/(member)/profile.tsx` — each badge in the list now has a star toggle. Tap to equip (auto-unequips the previous one), tap again to unequip. Disabled on expired badges.
- `apps/mobile/src/app/(member)/leaderboard.tsx` — each rank row now shows the equipped badge chip next to the member's name. Pure rendering change, no extra fetch per row (single map lookup).
- `apps/mobile/src/app/(member)/chat.tsx` — sender name row now shows the badge chip alongside the name (only for messages from others). Single shared map across all rendered messages; safe with the existing WebSocket message stream.
- `apps/mobile/src/types/index.ts` — `MemberBadge` gained `isEquipped`, added new types `EquippedBadgeDisplay` and `EquippedBadgeRow`.

### Key decisions
- **Exactly one badge equipped per member** (enforced in the transaction). Simpler UX (one star, clear pick), simpler rendering (one chip), avoids the "show which 3 of N" UI bikeshed.
- **Separate `/badges/equipped` endpoint** rather than embedding equipped badges in chat message payloads or leaderboard rows. Decouples render concerns, fits the existing WebSocket-driven chat (which we explicitly didn't migrate to TanStack Query), avoids re-shaping the leaderboard query that's used by other consumers.
- **60s staleTime on `useEquippedBadgesMap`.** Equipped badges change rarely; refetching aggressively would waste a join across the gym's member set. Mutation invalidates the cache immediately so the user sees their own change without delay.
- **Refused to touch chat message payload.** The "intentionally not migrated" status for chat (WebSocket + Tan-Stack incompatibility) still holds — surfacing badges via a separate map keeps that boundary intact.

### Deferred
- **Staff "award badge" mobile UI.** Endpoint already exists (`POST /badges/award`). Adding the screen is small, can land next session.
- **"Badge Gallery" tab on profile.** Current list section already shows everything. A dedicated tab can wait until catalog grows beyond ~10 badges per member.

### "Who's checked in" + privacy toggle (M)
Member-facing view of who's currently at the gym. Identity is **hidden by default**; members opt in via a switch on the screen itself.

#### API
- `db/migrations/0036_privacy_default_hidden.sql` — flip `member_privacy.hide_checkin_visibility` default from `false` to `true` and force-update all existing rows. The schema already had the column; no one had ever consented (no UI existed), so promoting everyone to hidden is the conservative, privacy-preserving default.
- `prisma/schema.prisma` — match the new default.
- `GET /gyms/:gymId/checkins/active-public` (new) — returns `{ totalCount, visible: [{ memberId, userId, fullName, checkedInAt }] }`. Filters server-side by `hideCheckinVisibility !== true`. Permission: gym access only (no `checkins.view` needed — every member sees this).
- `PATCH /gyms/:gymId/members/me/privacy` (new) — body `{ hideCheckinVisibility?, hideFromMemberList? }`. Upserts the privacy row and returns it.
- Use cases: `ListActivePublicCheckinsUseCase`, `UpdateMyPrivacyUseCase`.
- `MembersRepository.findFullMemberByUserId` now also selects `privacy` so the existing `getMyMember` response carries the current state — no extra round-trip on the client.
- `MembersRepository.upsertMemberPrivacy` (new) — handles the safety case where the privacy row is missing for a legacy member.
- Specs: `list-active-public-checkins.use-case.spec.ts` (4 cases) + `update-my-privacy.use-case.spec.ts` (3 cases). All passing.

#### Mobile
- Types: added `MemberPrivacy`, `PublicActiveCheckin`, `PublicActiveCheckinsResult`. `GymMember` gained an optional `privacy` field.
- `member.service.ts` — `getActivePublicCheckins`, `updateMyPrivacy`.
- `hooks/members.ts` — `useActivePublicCheckins()` (30s staleTime, refreshes naturally), `useUpdateMyPrivacy()` (patches the cached `me` query so the UI reflects the toggle instantly, then invalidates the public list).
- `apps/mobile/src/app/(member)/whos-here.tsx` (new) — full screen:
  - Top card: total count, with a sub-note for members hidden privately
  - Privacy toggle row with adaptive copy ("You're hidden…" vs "You're visible…")
  - Sorted list of visible members with relative time ("12 min ago") and an equipped badge chip beside the name
  - Pull-to-refresh; clean empty state distinguishing "no one is here" vs "no one is visible"
- `apps/mobile/src/app/(member)/dashboard.tsx` — added "X members are here right now" pressable card that taps through to the screen.

### Key decisions
- **Default hidden, opt-in to show.** Privacy is the right default; the migration retroactively flips every existing row.
- **Dashboard entry point** rather than living under the check-in screen — gives the feature higher visibility and surfaces the count even when you haven't tapped check-in.
- **Toggle lives on the screen itself**, not in a settings menu. Discoverable exactly where it matters; one screen owns both the data and the control.
- **Staff visibility unchanged.** Existing staff endpoints (`/checkins/active`) still return everyone. Privacy is a member-vs-member concept; staff still need to operate normally.
- **No separate column** — reused the pre-existing `member_privacy.hide_checkin_visibility` field (its purpose was always this; the comment in `0010_gym_members.sql` even spells it out).
