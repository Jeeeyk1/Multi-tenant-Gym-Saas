## 1. `GYM_OWNER` permission bug (root cause, not just a reseed)

### Symptom
Admin/owner accounts got `Missing permission: leaderboard.review` when reviewing leaderboard PR submissions.

### Root cause
`db/migrations/0029_gym_owner_role.sql` creates the `GYM_OWNER` role and tries to grant it every permission via `CROSS JOIN permissions`. But migrations run before seeds (`db:migrate && db:seed`), and permissions are defined in `db/seeds/002_permissions.sql`, not a migration. So at the moment 0029 ran, the `permissions` table was empty — the cross join inserted nothing. `GYM_OWNER` has had zero permissions since that migration was introduced, for every environment that ran it (confirmed via direct DB query: 0/22 permissions granted before the fix).

This wasn't leaderboard-specific — any `GYM_OWNER` account was silently missing every permission check.

### Fix
Added a `GYM_OWNER` grant to `db/seeds/003_role_permissions.sql` (which runs after permissions exist), granting it every permission unconditionally. This is durable: it self-heals on every fresh DB, CI run, and `pnpm db:seed` re-run, and stays correct as future permissions are added (the exact failure mode that bit `leaderboard.review`).

Verified before/after via direct query against `role_permissions`: `GYM_OWNER` went from 0/22 to 22/22 granted permissions.

### Files changed
- `db/seeds/003_role_permissions.sql`

### Action needed from the user
Any admin/owner session needs to log out and back in (or wait for the silent token refresh) to pick up the corrected permission set — permissions are baked into the JWT at login/refresh time, not read live per-request.

## 4. Check-ins trend graph

New feature, built end-to-end. Scoped with the user: web + mobile together, 7-day/30-day toggle.

### API
New `reports` module (the `reports.view` permission already existed in the seed but had no module behind it):
- `GET /gyms/:gymId/reports/checkins-trend?days=7|30` — zero-filled daily check-in counts, gated by `reports.view`.
- Repository uses a `generate_series` + `LEFT JOIN check_ins` raw query so days with no check-ins still appear as `0` (no precedent for `$queryRaw` elsewhere in the codebase, but needed here for zero-fill; verified directly against the dev DB).
- Follows the established feature-first module structure (`application/use-cases`, `infrastructure/persistence`, `presentation/{controllers,dto}`), same `assertGymAccess`/`assertPermission` pattern as every other use case in the codebase.

### Web
- `recharts` added as a new dependency.
- `lib/actions/reports.ts` — server action wrapping the API call (existing pattern, mirrors `lib/actions/leaderboard.ts`).
- `components/dashboard/checkins-trend-chart.tsx` — client component, bar chart with a 7d/30d toggle. Initial 7-day data is fetched server-side in `app/dashboard/page.tsx` (consistent with how the rest of that page already fetches); toggling to 30 days calls the server action via `useTransition`.
- Chart only renders when `user.permissions` includes `reports.view`.

### Mobile
- `react-native-chart-kit` added (react-native-svg was already a dependency).
- `services/reports.service.ts` + `hooks/reports.ts` — same shape as `services/leaderboard.service.ts` / `hooks/leaderboard.ts`.
- `components/CheckinsTrendChart.tsx` — bar chart with the same 7d/30d toggle, rendered on `(staff)/dashboard.tsx` only when the user has `reports.view`.
- Mobile staff screens are still on the static `COLORS` theming pattern (not yet migrated to the light/dark `theme.colors` pattern from 2026-06-29's session) — this component matches that existing convention rather than mixing patterns.
