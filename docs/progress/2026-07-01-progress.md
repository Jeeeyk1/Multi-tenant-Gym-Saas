#### Bug fixed: renewals gymId scoping
`MembershipRenewal` has no direct `gymId` column (it scopes through `GymMember`), but
`buildWhere` in `insights.repository.ts` was passing `{ gymId }` directly to the Prisma
renewals delegate. This would cause a Prisma runtime error on any revenue query. Fixed by
adding entity-specific gym scoping in `buildWhere`:
```
renewals → where: { member: { gymId } }
members/checkins → where: { gymId }
```

#### New: check-in method dimension + filter
`METRIC_ENTITIES.checkins` now exposes `method` as both a `dimension` (group_by) and a `filter`.
Answers: "how many QR-scan check-ins vs manual this month?", "which check-in method is most popular?"
Valid values: `MANUAL_STAFF`, `QR_STAFF_SCAN`, `QR_SELF_SCAN`, `APP_SELF_CHECKIN`.

#### New: `get_trend` tool (zero-filled time series)
Answers "per day/week/month" questions for all three entities including revenue. Architecture:
- `trend.whitelist.ts` — `TrendQuery` type + `validateTrendQuery` (entity/granularity/field
  validation, date-range defaulting: 30d/12w/12m by granularity)
- `runTrend()` in `insights.repository.ts` — three `$queryRaw` methods (checkins, members,
  renewals) using `generate_series(...) LEFT JOIN (...)` for zero-filling. Renewals revenue
  scopes through `JOIN gym_members gm ON gm.id = mr.member_id WHERE gm.gym_id = $gymId`.
- `tool-registry.ts` — new `get_trend` schema + executor wired, system prompt updated with
  check-in method values and trend tool description.

SQL verified directly against dev DB: checkins trend returns zero-filled 7-day series;
renewals revenue trend surfaces a real ₱1500 June data point with correct gym scoping.

Shared with `trend.whitelist.ts` to avoid duplicating the date-range parsing logic.

## Phase 1 — API core

A new admin-facing AI feature: gym owners/managers ask natural-language questions about their
gym answered from the database via tool calling. See Phase 1 notes in the previous session entry.

### Architecture decisions
- Tool calling, not text-to-SQL. `gymId` injected from JWT, never in tool input schema.
- `query_metric` structured tool over a whitelist — flexible, safe, grows without new code.
- `get_trend` tool for time series — zero-filled `generate_series` pattern.
- Model: Claude Haiku 4.5 via dedicated `ANTHROPIC_API_KEY`.
- Subscription gating via existing `ai_assistant` plan-feature + `aiTokenQuotaMonthly`.
- RBAC: `insights.query` permission → MANAGER + GYM_OWNER.
